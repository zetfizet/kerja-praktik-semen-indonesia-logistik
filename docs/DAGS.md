## Overview

The Airflow Stack includes **3 main DAGs** that handle different aspects of data orchestration:

| DAG | Purpose | Schedule | Status |\n|-----|---------|----------|--------|\n| `daily_warehouse_sync` | Daily warehouse data sync from DEVOM | Every day @ 00:00 UTC | ✅ Production |\n| `weather_data_fetch` | Real-time weather data collection | Every 1 hour | ✅ Production |\n| `sync_data_from_app` | Application database sync | On-demand | ✅ Development |\n\n---\n\n## 1. daily_warehouse_sync\n\n**Primary ETL pipeline for warehouse synchronization**\n\n### Basic Information\n\n```\nDAG ID:           daily_warehouse_sync\nOwner:            data_team\nSchedule:         0 0 * * *  (Daily @ 00:00 UTC = 7:00 AM WIB)\nRetry Policy:     2 retries with 5-minute interval\nCatchup:          Disabled\nTags:             warehouse, daily, sync\nStart Date:       January 21, 2026\n```\n\n### Purpose\n\nAutomatic daily synchronization of all data from the source DEVOM warehouse (devom.silog.co.id) to the local PostgreSQL warehouse:\n\n- **90+ business tables** across 8 schemas\n- **Weather data** (fact_weather_hourly, etc.)\n- **Analytics calculated fields**\n- **Full schema sync** (new tables detected automatically)\n\n### Data Sources\n\n#### Source Database: devom.silog.co.id\n\n```\nHost:       devom.silog.co.id\nPort:       5432 (default)\nDatabase:   om\nUser:       om\nPassword:   om\nProtocol:   PostgreSQL\n```\n\n#### Target Database: Warehouse (localhost:5433)\n\n```\nHost:       localhost\nPort:       5433\nDatabase:   warehouse\nUser:       postgres\nPassword:   postgres123 (dev credential)\nSchemas:    public (main), weather (fact tables), analytics (KPIs)\n```\n\n### Task Flow\n\n```\n┌─────────────────────┐\n│ check_weather_data  │  (1-2 min)\n│ - Verify BMKG API   │\n│ - Check freshness   │\n└──────────┬──────────┘\n           │\n           ▼\n┌─────────────────────┐\n│sync_warehouse_tables│  (5-15 min)\n│ - Connect to devom  │\n│ - List all tables   │\n│ - Batch sync data   │\n│ - Deduplication     │\n└──────────┬──────────┘\n           │\n           ▼\n┌──────────────────────┐\n│verify_warehouse_data │  (1-2 min)\n│ - Count validation   │\n│ - Schema check       │\n│ - Write summary logs │\n└──────────────────────┘\n```\n\n### Task Details\n\n#### Task 1: check_weather_data\n\n**Operator:** PythonOperator  \n**Duration:** 1-2 minutes  \n**Purpose:** Validate weather data source before sync\n\n```python\ndef check_weather_data():\n    # 1. Check BMKG API connectivity\n    # 2. Verify weather table exists in warehouse\n    # 3. Log data freshness metrics\n    # Returns: status (success/warning/error)\n```\n\n**Success Criteria:**\n- ✅ BMKG API is reachable\n- ✅ Previous weather data exists\n- ✅ Data freshness is acceptable (≤24 hours old)\n\n**Failure Handling:**\n- If BMKG unreachable: Log warning, continue sync\n- If no previous data: Log info, continue sync\n\n#### Task 2: sync_warehouse_tables\n\n**Operator:** PythonOperator  \n**Duration:** 5-15 minutes (varies with data volume)  \n**Purpose:** Main ETL pipeline - sync all data\n\n```python\ndef sync_warehouse_tables():\n    # 1. Connect to source (devom.silog.co.id)\n    # 2. List all tables in source database\n    # 3. For each table:\n    #    - Fetch complete data\n    #    - Apply deduplication (remove duplicates)\n    #    - Batch insert into warehouse (5000 rows/batch)\n    #    - Log table status\n    # 4. Return sync summary\n```\n\n**Processing Details:**\n- **Batch size:** 5,000 rows per insert\n- **Deduplication:** Remove exact duplicate rows\n- **Strategy:** Full sync (not incremental)\n- **Logging:** Each table gets status log\n\n**Failure Handling:**\n- Table-level retry: If a table fails, skip and continue others\n- DAG-level retry: If entire task fails, retry after 5 minutes\n\n#### Task 3: verify_warehouse_data\n\n**Operator:** PythonOperator  \n**Duration:** 1-2 minutes  \n**Purpose:** Validate sync results\n\n```python\ndef verify_warehouse_data():\n    # 1. Count rows in each table\n    # 2. Compare with source counts\n    # 3. Check data types match\n    # 4. Verify schema matches\n    # 5. Generate verification report\n```\n\n**Validation Checks:**\n- ✅ Row count matches between source and target\n- ✅ All columns exist in target\n- ✅ Data types are compatible\n- ✅ No NULL values in key columns\n\n**Output:**\n- Writes verification summary to logs\n- Metrics available in Airflow UI\n\n### Performance Metrics\n\n#### Typical Execution Times\n\n```\nBest case:      5 minutes (small data volume, fast network)\nAverage case:   10 minutes (typical production run)\nWorst case:     25 minutes (all 90+ tables, slow network)\n```\n\n#### Resource Usage\n\n```\nCPU:        Medium (sync is I/O bound, not CPU intensive)\nMemory:     Medium (batch processing, ~500MB-1GB)\nNetwork:    High (large data transfers)\nDisk I/O:   Medium (database writes)\n```\n\n#### Data Volume\n\n```\nTables synced:      90+\nEstimated rows:     500K-2M (varies by business)\nDaily volume:       10-100MB (typical)\nTotal warehouse:    2-5GB (with history)\n```\n\n### Configuration\n\n#### Default Arguments\n\n```python\ndefault_args = {\n    'owner': 'data_team',           # Responsible team\n    'retries': 2,                   # Retry failed tasks 2x\n    'retry_delay': timedelta(minutes=5),  # Wait 5 min between retries\n    'start_date': datetime(2026, 1, 21),  # First run date\n}\n```\n\n#### Scheduling\n\n```\nSchedule:       0 0 * * *       (Cron: daily at midnight UTC)\nTimezone:       UTC (default)\nWhen in WIB:    7:00 AM         (UTC+7)\nWhen in WITA:   8:00 AM         (UTC+8)\nWhen in WIT:    9:00 AM         (UTC+9)\n```\n\n### Code Location\n\n```\nFile:           airflow/dags/daily_warehouse_sync.py\nLines:          ~500 lines including table mappings\nDatabase:       PostgreSQL queries (psycopg2)\nImports:        DAG, PythonOperator, psycopg2\n```\n\n### Monitoring & Alerting\n\n#### In Airflow UI\n\n1. **DAGs view:** See last run status\n2. **DAG detail:** See task execution timeline\n3. **Task logs:** Debug individual task failures\n4. **Tree view:** Historical run status\n\n#### Metrics to Monitor\n\n```\n✅ DAG success rate       (target: 100%)\n✅ Average duration       (target: < 15 minutes)\n✅ Task retry rate        (target: < 5%)\n✅ Data freshness         (target: < 24 hours old)\n```\n\n#### Common Issues\n\n| Issue | Cause | Solution |\n|-------|-------|----------|\n| Network timeout | DEVOM unreachable | Check VPN, ping devom.silog.co.id |\n| Database lock | Long running query | Kill blocking query in source DB |\n| Disk full | Warehouse DB too large | Archive old data or expand storage |\n| Memory error | Data too large for batch | Reduce batch size in code |\n\n---\n\n## 2. weather_data_fetch\n\n**Real-time weather data collection and monitoring**\n\n### Basic Information\n\n```\nDAG ID:           weather_data_fetch\nOwner:            data_team\nSchedule:         0 * * * *  (Every 1 hour)\nRetry Policy:     2 retries with 5-minute interval\nCatchup:          Disabled\nTags:             weather, bmkg, api, realtime\nStart Date:       January 21, 2026\n```\n\n### Purpose\n\nCollect real-time weather data from **BMKG API** (Badan Meteorologi, Klimatologi, dan Geofisika) and:\n\n- Store in warehouse database\n- Track data freshness\n- Detect and skip duplicates\n- Maintain 30-day rolling window\n- Monitor forecast accuracy\n\n### Data Source: BMKG API\n\n#### BMKG Weather API\n\n```\nProvider:       BMKG (Indonesian Meteorological Agency)\nEndpoint:       https://api.bmkg.go.id/\nData Points:    Temperature, humidity, wind, precipitation\nUpdate Freq:    Every 2 hours\nCoverage:       Multiple Indonesian cities & locations\nFormat:         JSON\n```\n\n#### Available Locations\n\n```\nJakarta         (Major city, capital)\nBandung         (Central Java)\nSurabaya        (East Java)\nMedan           (North Sumatra)\nBali            (Island)\nMakassar        (South Sulawesi)\n[... more cities]\n```\n\n### Task Flow\n\n```\n┌──────────────────────┐\n│ fetch_weather_data   │  (30-60 sec)\n│ - Call BMKG API      │\n│ - Parse JSON response│\n│ - Format data        │\n└──────────┬───────────┘\n           │\n           ▼\n┌──────────────────────┐\n│ deduplicate_data     │  (20-30 sec)\n│ - Check existing     │\n│ - Skip if duplicate  │\n│ - Insert new records │\n└──────────┬───────────┘\n           │\n           ▼\n┌──────────────────────┐\n│ track_freshness      │  (10-20 sec)\n│ - Calculate age      │\n│ - Tag freshness      │\n│ - Log metrics        │\n└──────────┬───────────┘\n           │\n           ▼\n┌──────────────────────┐\n│ cleanup_old_data     │  (5-10 sec)\n│ - Delete >30 days    │\n│ - Vacuum table       │\n│ - Log removed count  │\n└──────────────────────┘\n```\n\n### Task Details\n\n#### Task 1: fetch_weather_data\n\n**Operator:** PythonOperator  \n**Duration:** 30-60 seconds  \n**Purpose:** Fetch latest weather data from BMKG\n\n```python\ndef fetch_weather_data():\n    # 1. Build API request for each location\n    # 2. Call BMKG API endpoint\n    # 3. Parse JSON response\n    # 4. Extract: temperature, humidity, wind, precipitation\n    # 5. Normalize timestamp (Asia/Jakarta)\n    # 6. Return list of weather records\n```\n\n**API Response Format:**\n```json\n{\n  \"location\": \"Jakarta\",\n  \"temperature\": 28.5,\n  \"humidity\": 72,\n  \"wind_speed\": 4.2,\n  \"precipitation\": 0.5,\n  \"timestamp\": \"2026-04-13 10:00:00+07:00\"\n}\n```\n\n**Error Handling:**\n- If API fails: Retry 2x with exponential backoff\n- If timeout: Skip and continue\n- Parse errors: Log and continue with partial data\n\n#### Task 2: deduplicate_data\n\n**Operator:** PythonOperator  \n**Duration:** 20-30 seconds  \n**Purpose:** Prevent duplicate weather records\n\n```python\ndef deduplicate_data():\n    # 1. Get previous records for same location/time\n    # 2. Compare with new data\n    # 3. If duplicate: Skip insertion\n    # 4. If new: Insert with INSERT OR IGNORE\n    # 5. Return count of inserted records\n```\n\n**Deduplication Logic:**\n- **Key fields:** location, timestamp (rounded to hour)\n- **Comparison:** Compare all numeric fields\n- **Action:** If 100% match, skip; if differs, insert as new record\n\n**Database Query:**\n```sql\nINSERT INTO weather.fact_weather_hourly\n  (location, temperature, humidity, wind_speed, precipitation, timestamp, created_at)\nVALUES (%s, %s, %s, %s, %s, %s, NOW())\nON CONFLICT (location, timestamp_hour) DO NOTHING;\n```\n\n#### Task 3: track_freshness\n\n**Operator:** PythonOperator  \n**Duration:** 10-20 seconds  \n**Purpose:** Monitor data freshness\n\n```python\ndef track_freshness():\n    # 1. Get latest record for each location\n    # 2. Calculate age in minutes\n    # 3. Classify: FRESH / WARNING / STALE\n    # 4. Store classification in database\n    # 5. Log metrics by category\n```\n\n**Freshness Classification:**\n\n```\nData Age        | Status      | Color | Action\n┌─────────────────────────────────────────────────┐\n≤ 60 minutes    | FRESH (✅)  | Green | OK, use data\n60-180 minutes  | WARNING (⚠️) | Yellow| Acceptable\n> 180 minutes   | STALE (❌)  | Red   | Alert, may be old\n```\n\n**Metrics Output:**\n```\nTotal records:      250+\nFresh (≤60 min):    240   (96%)\nWarning (60-180):   10    (4%)\nStale (>180):       0     (0%)\n```\n\n#### Task 4: cleanup_old_data\n\n**Operator:** PythonOperator  \n**Duration:** 5-10 seconds  \n**Purpose:** Maintain data storage efficiency\n\n```python\ndef cleanup_old_data():\n    # 1. Find records older than 30 days\n    # 2. Delete old records\n    # 3. Vacuum table (reclaim space)\n    # 4. Log count of deleted records\n```\n\n**Retention Policy:**\n```\nKeep last:      30 days\nDelete older:   >30 days\nVacuum freq:    Every run\nExpected rate:  ~1-2% of records deleted per run\n```\n\n**Cleanup Query:**\n```sql\nDELETE FROM weather.fact_weather_hourly\nWHERE created_at < NOW() - INTERVAL '30 days';\n\nVACUUM ANALYZE weather.fact_weather_hourly;\n```\n\n### Performance Metrics\n\n#### Typical Execution Times\n\n```\nBest case:      60 sec   (fast API, small dataset)\nAverage case:   2 min    (typical production)\nWorst case:     5 min    (API slow, large dataset)\n```\n\n#### API Rate Limits\n\n```\nRequests/hour:  Unlimited (as of last update)\nTimeout:        30 seconds per request\nRetry policy:   2x with 5-min backoff\nBackoff format: Exponential (5s, 25s, 2m)\n```\n\n#### Data Volume\n\n```\nRecords/hour:   50-100 (multiple locations)\nDaily records:  1,200-2,400\nMonthly:        36K-72K\nStorage:        ~50-100MB per year\n```\n\n### Configuration\n\n#### Default Arguments\n\n```python\ndefault_args = {\n    'owner': 'data_team',\n    'retries': 2,\n    'retry_delay': timedelta(minutes=5),\n    'start_date': datetime(2026, 1, 21),\n}\n```\n\n#### Scheduling\n\n```\nSchedule:       0 * * * *       (Cron: every hour)\nTimezone:       UTC\nActual runs:    00:00, 01:00, 02:00, ..., 23:00 UTC\nDuration:       ~2 minutes each\nDaily runs:     24 runs\n```\n\n### Monitoring & Alerting\n\n#### Metrics to Monitor\n\n```\n✅ API availability         (target: 99%+ uptime)\n✅ Data freshness           (target: 100% < 60 min)\n✅ Duplicate rate           (target: < 5%)\n✅ Task success rate        (target: 100%)\n✅ Average response time    (target: < 2 min)\n```\n\n#### Alert Conditions\n\n| Condition | Threshold | Action |\n|-----------|-----------|--------|\n| API unavailable | 2 consecutive failures | Alert ops team |\n| Data stale | >20% in STALE status | Check BMKG API |\n| Task failure | 2+ failures in 24h | Trigger alert |\n| Storage full | >80% disk used | Archive old data |\n\n---\n\n## 3. sync_data_from_app\n\n**Application database synchronization (Development/On-demand)**\n\n### Basic Information\n\n```\nDAG ID:           sync_data_from_app\nOwner:            data_team\nSchedule:         None (manual trigger only)\nRetry Policy:     2 retries\nStatus:           Development stage\nTags:             app, etl, manual\n```\n\n### Purpose\n\nSync selected tables from application database to Airflow internal database for ETL processing:\n\n- Bridge between app DB and warehouse\n- Prepare data for transformation\n- Support on-demand data operations\n- Enable data validation before warehouse sync\n\n### Source & Target\n\n#### Source: Application Database\n\n```\nHost:       devom.silog.co.id\nPort:       5432\nDatabase:   devom.silog.co.id\nUser:       om\nPassword:   om\nTables:     Selected app tables (driver, armada, etc.)\n```\n\n#### Target: Airflow Metadata Database\n\n```\nHost:       postgres (container)\nPort:       5432\nDatabase:   airflow\nUser:       airflow\nPassword:   airflow\nTables:     app_* (temporary staging)\n```\n\n### How to Trigger\n\n#### Method 1: Airflow UI\n\n1. Navigate to **DAGs** tab\n2. Find `sync_data_from_app`\n3. Click the **Trigger DAG** button\n4. Click **Trigger**\n\n#### Method 2: Airflow CLI\n\n```bash\nairflow dags trigger sync_data_from_app\n```\n\n#### Method 3: Airflow REST API\n\n```bash\ncurl -X POST http://localhost:8080/api/v1/dags/sync_data_from_app/dagRuns \\\n  -H \"Content-Type: application/json\" \\\n  -d '{\"conf\": {}}'\n```\n\n### Code Location\n\n```\nFile:       airflow/dags/sync_data_from_app.py\nPython:     ~300 lines\nORM:        Direct SQL (psycopg2)\n```\n\n---\n\n## DAG Configuration\n\n### Default Arguments (All DAGs)\n\nEvery DAG has these default arguments:\n\n```python\ndefault_args = {\n    'owner': 'data_team',              # Responsible team\n    'retries': 2,                      # Retry failed tasks 2x\n    'retry_delay': timedelta(minutes=5),   # Wait 5 min between retries\n    'start_date': datetime(2026, 1, 21),   # Historical start date\n    'catchup': False,                  # Don't run past dates\n}\n```\n\n### Execution Configuration\n\n```python\nDAG(\n    'dag_id',\n    default_args=default_args,\n    description='...',\n    schedule='...',                    # Cron schedule\n    catchup=False,                     # No backfilling\n    tags=['...'],                      # For filtering\n    max_active_runs=1,                 # One run at a time\n)\n```\n\n### Database Connections\n\nDAGs use these connections (configured in Airflow Admin):\n\n```\nConnection ID       Type        Host                Database\n─────────────────   ──────      ─────────────────   ──────────\npostgres_warehouse  PostgreSQL  localhost:5433      warehouse\npostgres_app        PostgreSQL  devom.silog.co.id   om\npostgres_airflow    PostgreSQL  postgres:5432       airflow\n```\n\n---\n\n## Error Handling & Retries\n\n### Retry Mechanism\n\n**Default retry behavior (all DAGs):**\n\n```python\nretries = 2                              # Try up to 3 times (original + 2 retries)\nretry_delay = timedelta(minutes=5)      # Wait 5 minutes between attempts\n```\n\n**Timeline for failed task:**\n\n```\nTime 0:00    Task fails (Attempt 1)\nTime 0:05    Wait 5 minutes\nTime 0:05    Retry (Attempt 2)\nTime 0:10    If fails again, wait 5 minutes\nTime 0:15    Retry (Attempt 3)\nTime 0:20    If fails again, mark task as FAILED\n```\n\n### Failure Scenarios\n\n#### Scenario 1: Network timeout (Retryable)\n\n```\n❌ Attempt 1: Network timeout to devom.silog.co.id\n   → Wait 5 min\n✅ Attempt 2: Connection succeeds\n```\n\n#### Scenario 2: Database locked (Self-healing)\n\n```\n❌ Attempt 1: \"database is locked\" error\n   → Wait 5 min (lock released)\n✅ Attempt 2: Lock acquired, query succeeds\n```\n\n#### Scenario 3: Invalid data (Not retryable)\n\n```\n❌ Attempt 1: \"Invalid UTF-8 in column X\"\n❌ Attempt 2: Same error (retries won't help)\n❌ Attempt 3: Same error\n   → Task marked FAILED, requires manual investigation\n```\n\n### Monitoring Task Failures\n\n**In Airflow UI:**\n\n1. Navigation **DAGs** → Click DAG name\n2. Look for red/orange boxes in task grid\n3. Click failed task\n4. Click **Log** tab\n5. Review error message\n6. Decide: retry or investigate\n\n**Common errors:**\n\n```\n\"could not connect to server\"      → Network issue\n\"permission denied\"                → Database access issue\n\"out of memory\"                    → Data too large\n\"unique constraint violation\"      → Duplicate data\n\"timeout after X seconds\"          → Slow query or network\n```\n\n### Manual Recovery\n\nIf a DAG fails and auto-retry doesn't help:\n\n#### Option 1: Trigger again immediately\n\n```bash\nairflow dags trigger daily_warehouse_sync\n```\n\n#### Option 2: Clear and retry failed task\n\n```bash\n# Clear failed task (in Airflow UI)\n# DAG → DAG Runs → Run ID → Task → Clear\n\n# Then trigger DAG again\nairflow dags trigger daily_warehouse_sync\n```\n\n#### Option 3: Clear entire DAG run\n\n```bash\n# Start over\nairflow dags clear daily_warehouse_sync --confirm\nairflow dags trigger daily_warehouse_sync\n```\n\n---\n\n📖 **Next:** Read [Architecture Guide](ARCHITECTURE.md) to understand system design  \n👈 **Back to:** [Main README](../README.md)\n
